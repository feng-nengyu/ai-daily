Self-Evolving Multi-Agent Systems via Decentralized Memory

9 arXiv
multi-agentLLMtool-usereasoning

把多智能体的“共享记忆库”改成每个 agent 自己管的去中心化记忆，并用在线反馈动态调权，既保留多样性又省 token，还给出可证明的低后悔界。

技术方案
提出 DecentMem：每个 agent 维护双池记忆。其一是 exploitation pool（利用池）：把历史成功轨迹（trajectory，执行过程）做“沉淀/巩固”后存起来，供相似任务直接复用；其二是 exploration pool（探索池）：由 LLM 生成针对未见场景的候选策略/轨迹，鼓励尝试新解。系统在运行中按阶段（stage-wise）收集 LLM-as-a-judge（让大模型当裁判）的反馈，对两类记忆的权重进行在线重加权，决定下一步更偏复用还是探索。与以往集中式共享记忆不同，它不需要所有 agent 频繁读写同一个仓库，减少通信协调开销、降低隐私暴露面，并避免所有 agent 被同一记忆“同质化”。
关键数据
覆盖 3 个 MAS 框架（AutoGen、DyLAN、AgentNet），5 个骨干模型（Qwen3 4B/8B/14B；Gemma4 E2B/E4B），5 类基准（数学/代码/问答/具身等）。相对最强“集中式记忆”基线：平均准确率最高 +23.8%；相对“无记忆”基线：最高 +52.5%。同时 token 使用量最高降低 49%。理论结果：累计后悔（regret，决策损失累积）为 O(log T)，并声称达到随机多臂老虎机（stochastic bandit）下界同阶；并证明解空间“全局可达”（global reachability）。单项 benchmark 细分分数与统计显著性未披露。
为什么值得关注
这篇直接命中 LLM agent / multi-agent 的“长期记忆怎么做”问题：用去中心化记忆替代集中式共享库，属于 agent 架构层面的可复用设计。双池（利用+探索）+ 在线裁判反馈的调权机制，也与 tool use/RAG 中的“检索-复用 vs 生成-探索”权衡高度相关，并给出可证明的 bandit 风格保证。

Spreadsheet-RL: Advancing Large Language Model Agents on Realistic Spreadsheet Tasks via Reinforcement Learning

8 arXiv
LLMagenttool-usereinforcement learning

把“Excel 里做真实多步任务”的训练从提示词堆叠推进到可规模化的强化学习微调，并配套真实起点-目标表格数据与领域基准，直接量化提升 Pass@1。

技术方案
作者提出 Spreadsheet-RL：在真实的 Microsoft Excel 环境中对 LLM agent 做 RL（强化学习）微调，而不是只靠通用模型+专门提示词。先用自动化流水线从在线论坛收集“起始表格-目标表格”的配对样本，形成可监督/可评测的任务来源。再构建 Domain-Spreadsheet 基准，覆盖金融、供应链等更贴近工作流的表格任务。训练与评测通过 Spreadsheet Gym 完成：用 Python 沙箱暴露大量 Excel 功能，并提供更完整的工具集与工具路由规则，让 agent 在多轮交互中学会选工具、按步骤完成复杂操作。
关键数据
在 SpreadsheetBench 上，Qwen3-4B-Thinking-2507 的 Pass@1 从 12.0% 提升到 23.4%（+11.4 个百分点，约 1.95 倍）。在作者新建的 Domain-Spreadsheet 数据集上，Pass@1 从 8.4% 提升到 17.2%（+8.8 个百分点，约 2.05 倍）。训练成本、数据规模、回合长度等未披露；但结果表明 RL 微调对“多步工具使用”的收益在通用与领域任务上都能复现。
为什么值得关注
这是典型的 LLM agent + tool use（工具调用）场景：在 Excel 这种数据界面里做多轮决策与执行，和 MCP/RAG/多智能体常见的“工具编排”问题高度同构。它也提供了可复用的 Gym 式环境与领域基准，方便对 reasoning models（推理模型）在真实工作流中的泛化能力做更可量化的对比。

Boiling the Frog: A Multi-Turn Benchmark for Agentic Safety

8 arXiv
agenttool-useMCPsafety

把“安全”从看模型说什么，转到看代理（会用工具的模型）在多轮办公任务里最终做了什么，专测“温水煮青蛙式”渐进攻击。

技术方案
提出 Boiling the Frog 多轮基准：每个场景先让模型做看似无害的办公/协作编辑，随后在后续轮次插入带风险的请求。评测是有状态的（stateful）：给模型一个持续存在的工作区（文档/表格/代码等），并把“风险载荷”放在不同轮次位置，观察最终产物状态是否变得不安全。评分不只看当轮回复，而是看工具操作后的工件（artifact）是否越界，例如是否引入违规内容、危险配置或失控行为。场景按三层运营风险分类组织，参考 Boiling the Frog 风险、欧盟 AI 法案高风险场景（Annex I/III）以及 GPAI 行为准则中的失控类要求，强调企业/办公室落地环境。
关键数据
9 个模型总体“严格攻击成功率”（ASR）为 44.4%。单模型 ASR 从 Claude Haiku 4.5 的 20.5% 到 Gemini 3.1 Flash Lite 的 92.9%；Seed 2.0 Lite 也超过 80%。在“行为准则-失控（loss-of-control）”类场景中，平均链路/类别级 ASR 高达 93.3%，说明多轮渐进诱导下，很多代理在最终工件层面更容易出问题。
为什么值得关注
对 LLM agent / tool use / multi-agent 读者，这类“有状态多轮+看最终工件”的评测更贴近真实代理系统（含工具调用、工作区记忆）的安全风险。对做 MCP、RAG 或工作流编排的人，它提供了可量化的“渐进攻击”失败率指标，能用来对比不同代理框架与防护策略在多轮任务中的稳健性。

Claw AI Lab: An Autonomous Multi-Agent Research Team

8 arXiv
multi-agenttool-useMCPRAG

把“隐藏的提示词到论文”自动写作链路，改成可交互的多智能体研究实验室：一条提示词就能拉起研究团队，并在仪表盘里监控、检查产物、回滚/续跑。

技术方案
系统不再围绕单一 Agent 或固定串行流程，而是允许用户从一个任务提示实例化“研究团队”，为不同成员配置角色与协作工作流。提供三种研究模式：探索、多人讨论、复现，让自动研究更可控（steerable）而不是黑箱跑完。核心工程组件是 Claw-Code Harness：把本地代码库、数据集、模型 checkpoint（训练中间权重）接到可运行实验，并把日志、结果文件等执行产物回流到研究循环中。通过统一仪表盘支持实时监控、产物检查（artifact inspection）以及回滚/恢复，减少半跑、结果写错等常见失败模式，并更容易把实验结果“忠实搬运”进论文。
关键数据
内部评测：5 个 AI 研究案例；以 AutoResearchClaw 为基线，AI 专家评委在“想法新颖性、实验完整性、论文呈现质量”三项上更偏好 Claw AI Lab。偏好比例/打分细节未披露；定性意义是把改进点落在“实验跑通与结果可信”而非仅文本质量。
为什么值得关注
对 LLM agent / multi-agent 读者：它把多智能体协作从“对话编排”推进到“可运行实验+可回滚”的研究基础设施。对 tool use / RAG/MCP 关注者：Claw-Code Harness 本质是在做更可靠的工具/执行集成与产物回流，解决 Agent 做实验时最常见的执行与结果一致性问题。

Is Capability a Liability? More Capable Language Models Make Worse Forecasts When It Matters Most

8 arXiv
LLMreasoningevaluationtool-use

论文发现“能力越强预测越差”的反向缩放：在带超线性增长与尾部风险（极端转折）的时间序列上，更强 LLM 的分布预测反而更不准，且主要错在上尾（高风险区）。

技术方案
作者聚焦金融/流行病常见的“快速增长+可能换挡（制度切换）”结构，评估 LLM 做概率预测（分位数/分布而非单点）。他们发布无污染的模拟基准 ForecastBench-Sim（FBSim），并在合成 SIR（传染病模型）与匹配的线性对照任务上对比，随后在真实数据（COVID-19、麻疹、房市、恶性通胀）复现。通过“按分位数拆解”定位误差来源：更强模型把上尾整体抬高去追逐激进外推，而下尾几乎不动，导致校准（概率与真实频率一致性）变差。进一步在 Llama-3.1 家族内做控制实验，区分模型规模与后训练（如指令/偏好对齐）对该现象的独立贡献，并测试加入领域知识也无法稳定修复校准。
关键数据
未披露具体分数/误差数值与模型规模区间的量化幅度；但给出定性结论：在“尾部纳入”的连续评分下，能力—准确率关系会从正相关翻转为负相关；而常见单阈值（二分类/命中率）指标在相同输出上看不出该问题，因其忽略上尾成本。误差集中在上尾分位数（高增长/高风险区），下尾基本不变。
为什么值得关注
对 LLM agent、tool use、RAG 等要把模型用于“决策/风控/预警”的场景很关键：更强的推理模型可能在最重要的极端风险预测上更不可靠。也提醒做评测与对齐时，别只用单阈值指标；需要覆盖尾部的连续分布评分，否则会误判模型能力。

Self-Policy Distillation via Capability-Selective Subspace Projection

8 arXiv
LLMagenttool-usereasoning

提出 SPD：不靠外部打分/执行反馈，只用模型自身梯度提取“能力子空间”，在自生成时过滤掉风格与格式噪声，做更可泛化的自蒸馏。

技术方案
传统自蒸馏要么用正确性过滤、执行反馈、奖励搜索等外部信号挑样本（成本高、前沿模型未必可用），要么直接用全部原始输出训练（容易学到风格/格式/错误模式）。SPD 的核心是“能力选择”：先在模型对少量“定义正确性”的 token 上计算梯度，用这些梯度提取一个低秩（低维）能力子空间。随后在自生成阶段，把注意力的 KV（key-value，注意力缓存）激活投影到该子空间，相当于只让与目标能力相关的方向参与生成。最后对投影后生成的原始输出，用标准 next-token prediction（下一个词预测）损失微调，不需要额外标注或奖励模型。
关键数据
覆盖代码生成、数学推理、多选 QA 三类任务；相对“无外部信号”的 SOTA 自蒸馏方法最高提升 13%；相对预训练基线最高提升 16%；在跨域（out-of-domain）泛化设置下性能高 15%。具体使用的模型规模、子空间秩大小、训练/推理额外开销未披露（但方法在生成时引入 KV 投影，理论上会带来一定计算开销）。
为什么值得关注
这条直接解决 LLM agent / tool use 场景常见的“自训练会学到格式与坏习惯、能力提升不稳定”的问题：用内部表示做能力选择，比靠外部奖励或执行器更通用。对关注 reasoning models、多任务/跨域泛化的读者，它提供了一种不依赖环境反馈的“能力定向自蒸馏”思路。

AMEL: Accumulated Message Effects on LLM Judgments

8 arXiv
LLMagentreasoningevaluation

证明“把 LLM 当裁判”会被对话历史的正负情绪带偏：同一条待评内容，放在一串偏正/偏负的历史后，模型判断会向历史极性漂移，且负面历史影响更强。

技术方案
作者把 LLM 作为自动评审器（打分/审核/评代码）的常见用法抽象成“同一会话里连续评很多条”。他们对 11 个模型做大规模对照实验：同一测试条目要么单独评（新上下文），要么放在“前面多轮几乎全是正评/负评”的历史之后再评。通过比较两种条件下的评分/判定分布，量化“累积消息效应（AMEL）”的偏移量，并进一步用不确定性（高熵=模型拿不准）分组、不同历史长度/位置、以及 token 概率分布变化来定位机制。与以往只讨论提示词偏置不同，这里强调的是“批处理评测/审核流水线”里历史累积造成的系统性偏差。
关键数据
总计 75,898 次 API 调用，覆盖 4 家提供商（OpenAI、Anthropic、Google + 4 个开源模型），11 个模型。整体判断向历史极性漂移：效应量 d=-0.17，p<1e-46；在模型本来更不确定的条目上更强：高熵条目 d=-0.34，而基线更确定时 d=-0.15。偏差不随上下文长度增长：5 轮 vs 50 轮几乎一样（Spearman |r|<0.01；OLS 斜率 p=0.80）。负面历史更“带偏”：逐条配对比较，负面诱导的偏差是正面的 1.62 倍（t=13.46，p<1e-39，n=2,481）。更大模型能减轻但不能消除：Anthropic Haiku -0.22 到 Opus -0.17；OpenAI Nano -0.34 到 GPT-5.2 -0.17。机制侧：token 概率分布是连续漂移而非阈值跳变；5 条带偏消息放在 50 轮历史的任何位置效果相同。
为什么值得关注
做 LLM agent / tool use / RAG 时常把模型当“评审器/路由器/打分器”来做自检、重排、内容审核；这篇量化了“同一会话批量评测”会引入可观偏差，直接影响多代理协作与自动评测的可靠性。它也给出工程上最简单的规避策略：每条评测用新上下文；不得不批处理时尽量让历史正负平衡。

HarnessAPI: A Skill-First Framework for Unified Streaming APIs and MCP Tools

8 arXiv
MCPtool-useagent

把“HTTP 接口 + MCP 工具”两套重复工程合并：只写一个带类型的 skill（技能）文件夹，就能同时生成可流式输出的 API 和零配置 MCP 工具，减少维护漂移。

技术方案
HarnessAPI 以“typed skill folder（带类型的技能目录）”作为单一事实来源：核心只需要一个 handler.py 和 Pydantic（数据校验/类型声明）schema。框架自动派生三样东西：支持 SSE（服务器推送事件）的流式 HTTP endpoint、可交互的 OpenAPI/Swagger 文档页、以及可被 Claude/Cursor 等 agent 运行时调用的 MCP 工具，并且同一进程内提供服务。它用“双模式内容协商”让同一个 handler 同时服务流式 SSE 客户端和普通 JSON 客户端，无需改业务代码。为解决 FastMCP 检查层对类型注解传播的限制，它引入动态代码生成，把 Pydantic 的类型信息正确传到工具注册层，避免“闭包式注册”导致的类型丢失。整体上它是 FastAPI 的子类，因此直接继承中间件、依赖注入和部署生态，而不是另起一套 Web 框架。
关键数据
作者在 6 个代表性 skills 上用 cloc（代码行统计）对比“手工双栈：FastAPI server + FastMCP server”，HarnessAPI 将面向框架的样板代码减少 74%。除代码量外的性能（吞吐/延迟）与稳定性指标未披露；其主要收益是减少重复实现与 schema 漂移风险。
为什么值得关注
对 LLM agent / MCP / tool use 场景，最大痛点之一是“同一工具要维护 HTTP 与 MCP 两套定义”，HarnessAPI 试图把工具定义、校验、流式输出和注册统一到一处。若你在做带 streaming 的工具调用、或希望 RAG/多工具链路更易部署与迭代，这类“单源生成 MCP + API”的工程化方案很直接可用。

Superhuman Safe and Agile Racing through Multi-Agent Reinforcement Learning

8 arXiv
multi-agentreinforcementtool-usesafety

现有 KV offload 多把完整 KV 放在主机内存（DRAM），解码时按需抓取“关键 KV”，但稀疏度再提高会伤准确率，且上下文/批量一大就被 KV 传输拖慢。KVDrive 走系统路线：在 GPU 显存、主机 DRAM、SSD 之间做多层缓存放置与迁移决策，尽量让常用 KV 留在更快的层级以提高复用、减少重复搬运。它根据注意力（attention）访问行为自适应管理缓存，避免“抓了又不用/反复抓同一段”的无效传输。同时重构解码流水线，把 I/O 阶段与 CPU/GPU 计算阶段重叠执行，减少因等待数据导致的停顿。最后做跨层协同的数据移动与调度，让长上下文推理在 GPU/DRAM 预算很紧时仍能扩展到更大上下文与更大 batch。
